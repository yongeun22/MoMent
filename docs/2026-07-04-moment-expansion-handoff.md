사용자의 핵심 요구는 다음과 같다.

- 기존 MoMent 사이트의 느낌을 최대한 유지한다.

- 요청이 불명확하면 추론으로 바로 실행하지 말고 이해한 내용을 먼저 확인한다.

## 핵심 디자인 방향

이 확장은 기능 추가보다 "기존 감상 경험 보존"이 우선이다.

유지해야 할 인상:

- 첫 화면은 여전히 사진 중심이어야 한다.
- 상단 메뉴는 얇고 조용해야 한다.
- 필터, 지도, 방명록 탭은 필요한 순간에만 나타나야 한다.
- 기능이 많아져도 첫 인상은 "검은 배경의 미니멀 온라인 사진 아카이브"여야 한다.

구체적 UI 원칙:

- 필터는 항상 펼쳐두지 말고 상단의 `필터` 항목을 눌렀을 때 열리는 패널로 둔다.
- 지도는 첫 로딩에 노출하지 않고 `지도` 메뉴에서만 진입한다.
- 라이트박스는 기본적으로 사진과 메타데이터 중심으로 둔다.
- 사진별 방명록은 기본 화면에 늘어놓지 않고 라이트박스 하단의 탭으로 숨긴다.
- 새 버튼, 탭, 입력창은 기존 방명록 팝업 스타일을 재사용한다.
- 모바일에서 라이트박스 이미지가 댓글 UI 때문에 지나치게 작아지지 않게 한다.

## 큰 구현 순서

### 1단계: 사진 데이터 기반 정리

먼저 사진 데이터 구조를 정리한다. 필터, 지도, 사진별 방명록은 모두 이 기반을 공유해야 한다.

기존 필드 유지:

```js
{
  id,
  imageUrl,
  lightboxUrl,
  originalName,
  date,
  location,
  photographer,
  createdAt,
  updatedAt
}
```

추가할 필드:

```js
{
  year,
  region,
  category,
  placeId,
  locationName,
  lat,
  lng,
  description
}
```

필드 의미:

- `year`: `"2025"` 같은 문자열. 날짜 필터용.
- `region`: `"경주권"`, `"서울·경기권"` 같은 권역.
- `category`: 여러 주제를 담을 수 있는 문자열 배열.
- `placeId`: 같은 장소를 안정적으로 묶기 위한 내부 id.
- `locationName`: 공개 표시용 장소명.
- `lat`, `lng`: 지도용 좌표. 좌표가 없는 사진은 지도에서 제외하되 갤러리에는 계속 표시.
- `description`: 향후 사진 설명 확장을 위한 선택 필드.

중요 판단:

- 지도에서 같은 장소를 묶을 때 `location` 문자열만 기준으로 삼지 않는다.
- `placeId`를 둬야 `"서울 경복궁"`, `"경복궁"`, `"경복궁 집옥재"` 같은 표기 차이를 제어할 수 있다.
- 기존 `location`은 호환용으로 유지하고, 새 구조에서는 `locationName`을 주 표시명으로 삼는다.

관리자 화면에도 위 필드를 입력/수정할 수 있게 한다.

### 2단계: 갤러리 필터

필터는 사진 데이터를 기반으로 한다.

필터 축:

- 연도: 전체, 2023, 2024, 2025, 2026
- 권역: 전체, 경주권, 서울·경기권, 충청권, 전라권, 경상권, 강원권, 제주권, 해외, 기타
- 주제: 전체, 사찰, 궁궐, 왕릉·고분, 박물관, 유물, 건축, 풍경, 의례·행사, 답사기록, 일상 속 문화유산
- 촬영자: 전체 + 실제 사진 데이터에서 추출한 촬영자 목록
- 촬영장소: 전체 + 실제 사진 데이터에서 추출한 장소 목록

사용자가 추가로 제안한 내용:

- 기존 카테고리에 촬영자와 촬영장소도 추가하고 싶다.
- 단, 설계상 촬영자/촬영장소는 "주제 카테고리"에 섞기보다 별도 필터 축으로 두는 편이 낫다.

동작:

- 여러 필터를 조합할 수 있어야 한다.
- 현재 표시 중인 사진 수를 보여준다.
- 필터 초기화 버튼을 둔다.
- 필터 결과가 없으면 조용한 빈 상태 메시지를 보여준다.
- 기존 모자이크 레이아웃은 유지한다.

UI 방향:

- 상단 메뉴에 `필터` 항목을 추가한다.
- 클릭하면 접히는 패널이 열린다.
- 패널은 기존 검은 반투명 overlay/card 톤을 따른다.
- 촬영장소는 항목이 많아질 수 있으므로 단순 버튼 나열보다 select 또는 검색 가능한 목록이 적절하다.

### 3단계: 방명록/사진 댓글 통합

기존 방명록을 일반 방명록과 사진별 방명록을 모두 담는 구조로 확장한다.

사용자 선호 표현:

- "댓글"보다 사이트 톤에 맞춰 "사진 방명록"이라고 부르는 편이 자연스럽다.
- 전체 메뉴명은 기존처럼 `방명록`을 유지한다.

새 방명록 object 기준:

```js
{
  id: "comment_001",
  type: "photo",
  photoId: "photo_20251010_gyeongbokgung_001",
  affiliation: "동국대",
  name: "신용은",
  text: "구도가 좋네요.",
  createdAt: "2026-07-04T12:30:00+09:00"
}
```

일반 방명록:

```js
{
  id: "guestbook_001",
  type: "general",
  photoId: null,
  affiliation: "동국대",
  name: "신용은",
  text: "잘 보고 갑니다.",
  createdAt: "2026-07-04T12:30:00+09:00"
}
```

마이그레이션 방향:

- 기존 `guestbook_entries`는 `type: general`, `photoId: null`로 취급한다.
- 기존 항목에는 `text`가 없으므로 빈 문자열 또는 호환 표시 정책이 필요하다.
- local Python DB와 Cloudflare D1 schema를 함께 확장해야 한다.

입력 필드:

- 소속
- 이름
- 내용

안전 처리:

- XSS 방지: 렌더링 시 escape 유지.
- 길이 제한: 소속, 이름, 내용 각각 제한.
- 도배 방지: 기존 rate limit 유지/확장.
- 금칙어/삭제 목록: 기존 moderation 로직 유지/확장.
- 관리자 삭제 기능: 일반 방명록과 사진 방명록 모두 삭제 가능해야 한다.

### 4단계: 사진 라이트박스 방명록 탭

사용자 제안:

- 방명록이 달린 사진에서도 방명록 탭을 추가한다.
- 해당 사진에 달린 방명록을 볼 수 있게 한다.
- `N개의 방명록이 달렸습니다.`처럼 수를 보여준다.

구현 방향:

- 라이트박스 하단에 탭을 추가한다.
- 탭 구성:
  - `정보`
  - `방명록 N`
- 기본 선택 탭은 `정보`다.
- `방명록 N` 탭을 누르면 해당 사진에 달린 방명록 목록과 작성 폼을 보여준다.
- 문구는 `이 사진에 N개의 방명록이 달렸습니다.`를 사용한다.
- 사진 방명록 작성 후:
  - 현재 라이트박스의 `방명록 N` 수가 즉시 갱신된다.
  - 전체 방명록 팝업에도 같은 항목이 반영된다.

모바일 주의:

- 라이트박스 이미지가 댓글 폼 때문에 작아지면 안 된다.
- 모바일에서는 이미지 아래 영역이 스크롤되도록 분리한다.
- 입력 폼은 기존 방명록 폼 스타일을 재사용하되 높이를 과하게 차지하지 않게 한다.

### 5단계: 전체 방명록 팝업 개선

기존 상단 `방명록` 메뉴는 유지한다.

추가 필터:

- 전체
- 일반 방명록
- 사진 방명록

사진 방명록 표시:

- 왼쪽에 해당 사진 썸네일을 작게 표시한다.
- 썸네일 클릭 시 해당 사진 라이트박스가 열린다.
- 항목에는 소속, 이름, 내용, 작성일, 연결된 장소명을 표시한다.

일반 방명록 표시:

- MoMent 기본 아이콘 또는 빈 썸네일 영역을 사용한다.
- 사진 방명록과 리스트 정렬이 크게 흔들리지 않게 한다.

톤:

- 방명록 팝업이 게시판처럼 무거워지면 안 된다.
- 현재의 작고 조용한 trace overlay 느낌을 유지한다.

### 6단계: MoMent Map / 답사지도 v1

지도는 가장 마지막에 구현한다.

이유:

- 지도 라이브러리, 좌표 데이터, 모바일 성능, 팝업 UI가 모두 추가되어 기존 갤러리 감상을 무겁게 만들 수 있다.
- 지도는 사진 감상 흐름의 보조 기능이어야 한다.

기본 방향:

- 상단 메뉴에 `지도` 항목을 추가한다.
- 지도는 사용자가 `지도`를 열 때만 로드한다.
- API 키가 필요 없는 Leaflet + OpenStreetMap 계열을 기본 후보로 둔다.
- 단, production dependency 추가가 필요하면 문서화하고 신중히 결정한다.

지도 데이터:

- 사진의 `placeId`, `locationName`, `region`, `category`, `lat`, `lng`를 사용한다.
- 핀은 사진 단위가 아니라 장소 단위로 표시한다.
- 같은 장소의 사진이 여러 장이면 핀 하나로 묶는다.

핀 팝업:

- 장소명
- 사진 수
- 대표 썸네일 1~3장
- `사진 보기` 버튼

상호 이동:

- 지도 썸네일 클릭 시 기존 사진 라이트박스가 열린다.
- `사진 보기` 클릭 시 해당 장소 사진만 갤러리에 필터링하거나 하단 패널로 표시한다.
- 라이트박스의 장소 메타데이터 옆에 `지도에서 보기` 버튼을 추가한다.
- `지도에서 보기` 클릭 시 지도 화면으로 이동하고 해당 장소 핀을 확대 표시한다.

### 7단계: 상호 이동 개선

최종적으로 다음 이동 흐름을 연결한다.

- 갤러리 필터 -> 사진 라이트박스
- 사진 라이트박스 -> 사진 방명록 탭
- 전체 방명록 팝업의 사진 썸네일 -> 사진 라이트박스
- 사진 라이트박스의 `지도에서 보기` -> 지도 핀
- 지도 핀의 `사진 보기` -> 해당 장소 사진 필터
- 필터 초기화 -> 전체 갤러리 복귀

## 구현 시 모듈화 제안

한 파일에 몰아넣지 않는다.

프론트엔드 JS는 기능별 분리를 권장한다.

예상 모듈:

- gallery rendering/filter state
- lightbox state and rendering
- guestbook API/client rendering
- map view and place grouping
- shared escaping/formatting helpers

현재는 `static/js/exhibition.js`에 대부분의 공개 사이트 로직이 들어 있다. 확장 구현 시 이 파일이 과도하게 커질 수 있으므로, 가능한 경우 기능 단위 모듈로 나누고 `index.html`에서 필요한 스크립트 로딩 방식을 정리한다.

백엔드도 다음 관심사를 분리한다.

- photo schema and serialization
- guestbook/comment validation
- guestbook/comment storage
- public payload export
- Cloudflare Functions schema compatibility

## 수용 기준

기능 구현 후 최소한 다음이 만족되어야 한다.

- 기존 갤러리 첫 화면의 시각적 느낌이 크게 달라지지 않는다.
- 입장 overlay, 모자이크 갤러리, hover/tap 메타, 라이트박스 확대가 유지된다.
- 필터를 닫은 상태에서는 기존 감상 경험과 거의 동일하다.
- 사진별 방명록 탭은 기본으로 펼쳐져 있지 않다.
- 전체 방명록과 사진 방명록이 같은 저장소/API 흐름에서 일관되게 동작한다.
- 사진 방명록이 달린 항목은 전체 방명록에서도 썸네일과 함께 표시된다.
- 지도 기능이 갤러리 초기 로딩을 무겁게 만들지 않는다.
- 모바일에서 상단 메뉴, 필터 패널, 라이트박스, 방명록 입력, 지도 팝업이 겹치지 않는다.
- static export 후 `dist/` 기준으로 공개 사이트가 동작한다.
- local Python server와 Cloudflare Pages Functions API의 데이터 형태가 서로 어긋나지 않는다.

## 테스트 계획

실행할 기본 테스트:

```powershell
py -3.13 -m unittest discover -s tests
py -3.13 manage.py export-static
```

검증 시나리오:

- 사진 등록 시 새 메타데이터가 저장된다.
- 사진 수정 시 새 메타데이터가 유지/갱신된다.
- export된 `data/photos.json`에 새 필드가 포함된다.
- 연도/권역/주제/촬영자/촬영장소 필터가 조합 동작한다.
- 필터 초기화가 전체 갤러리로 되돌린다.
- 라이트박스 메타데이터와 기존 이미지 확대가 깨지지 않는다.
- 일반 방명록 작성이 동작한다.
- 사진 방명록 작성이 특정 `photoId`와 연결된다.
- 사진 방명록 작성 후 해당 사진 라이트박스의 `방명록 N` 수가 갱신된다.
- 전체 방명록에서 일반/사진 방명록 필터가 동작한다.
- 전체 방명록의 사진 썸네일 클릭 시 해당 라이트박스가 열린다.
- XSS 문자열, 빈 입력, 너무 긴 입력, 반복 제출이 적절히 차단된다.
- 좌표가 있는 장소만 지도 핀으로 표시된다.
- 좌표 없는 사진은 지도에서는 제외되지만 갤러리에서는 정상 표시된다.
- `지도에서 보기`와 지도 팝업의 `사진 보기` 흐름이 동작한다.

## 남은 결정 사항

구현 전에 사용자에게 확인하면 좋은 사안:

- 사진별 댓글 표시명을 최종적으로 `방명록`, `사진 방명록`, `댓글` 중 무엇으로 할지.
- 기존 방명록 항목에 내용이 없는 상태를 어떻게 표시할지.
- 촬영장소 필터가 길어질 때 select 방식으로 갈지, 검색 가능한 패널로 갈지.
- 지도 라이브러리로 Leaflet을 추가해도 되는지.
- 좌표 입력을 관리자 화면에서 직접 입력할지, 장소별 사전 데이터로 관리할지.
- `placeId`를 자동 생성할지, 관리자 입력/수정 가능하게 할지.

현재까지의 기본 추천:

- 공개 표시명은 `방명록`과 `사진 방명록`을 사용한다.
- 필터 UI는 접이식 패널로 만든다.
- 지도는 마지막 단계에서 lazy load 한다.
- `placeId`는 내부적으로 안정적인 문자열을 두고, 관리자 화면에서는 장소명/좌표를 관리할 수 있게 한다.
- 기존 디자인과 감상 흐름을 건드리는 변경은 구현 중간마다 실제 화면 확인 후 조정한다.
